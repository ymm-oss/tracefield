**前提と検証順序**

まず潰す前提は、「FSLは正しさを証明する道具として売れる」という前提である。DOGFOOD-9/10/11で繰り返し出た事実は、内部整合だけでは意図ずれを閉じられず、境界ミスやguard weakeningは独立したforbidden / acceptance traceでしか捕捉されず、refund経路のような死経路はreachableで初めて露出した、というものだった。したがって価値仮説は「仕様の正しさ」ではなく、「要件ID付きの反例・死経路・禁止経路・境界欠落を、人間の判断可能な形で出す」へ固定する。

検証順序は次にする。第一に、方向Aで「LLMが自然文を読み直して修正するより、fslc JSON反例を使ったwrite→verify→repairの方が、修復率・局所化・再発防止を改善するか」を実案件相当ベンチで測る。第二に、方向Bで「生JSONや保証限界ではなく、PM/開発統制/内部監査が承認・差戻し・リスク受容に使える証跡へ翻訳できるか」を検証する。第三に、CIゲート単独ではなく、要件責任者が承認する監査・repair基盤として成立するかを検証する。第四に、支払主体、責任部署、非技術者向け証跡が明確でない施策は投資しない。エージェント間仕様交渉プロトコルは需要未証明なので本命にしない。

**方向A：AI仕様エージェントの検証OS**

方向Aの狙いは、FSLを人間が学ぶDSLではなく、LLMが仕様を書き、`fslc`が機械可読JSON反例を返し、Claude Code Skill / CI / PR差戻しがrepairを回す検証OSにすること。売り先は個々の開発者ではなく、生成AI開発基盤を持つ開発組織、Platform Engineering、DevEx、AI導入推進、開発統制である。支払主体は開発生産性予算またはAI開発基盤予算、責任部署はPlatform/DevExと開発統制の共同責任に置く。非技術者向け証跡は、要件ID別に「何が破れたか」「どの反例で再現するか」「修復後に何が閉じたか」「残る保証限界は何か」を示すPR添付レポートにする。

最初の施策は、実案件相当ベンチの作成である。入力は自然文要件、既存実装ログ、受入trace、forbidden trace、reachable期待、SLA条件、要件ID registryを含む。タスクは、LLMにFSLを書かせ、`fslc`のBMC+k帰納法、refinement、scenarios、実装ログ適合チェックを回し、JSON反例を使ってrepairさせる。比較対象は、自然文だけを再読させて修正するLLMフロー。測定値は、初回検出率、修復率、反例から修正箇所までの局所化精度、同種ミス再発率、要件ID欠落検出率、forbidden欠落率、reachable死経路検出率にする。

この施策が効く因果は、DOGFOOD-10で境界ミス/guard weakeningが独立traceなしでは捕捉されなかったため、LLMに「何となく直せ」と言うより、要件ID付きの反例JSONを返して修正対象を限定した方が修復が局所化するからである。DOGFOOD-9のrefund死経路は、安全性だけ通っても機能経路が死ぬことを示しており、reachableをrepairループに入れない限り「通った仕様」が実務上の欠落を隠す。DOGFOOD-9/10/11の意図ずれは、LLM内部の再解釈では閉じないため、acceptance/forbidden/reachableを外部証拠として固定する必要がある。

順序は、まず20〜30件の小型ベンチで修復率と局所化を測る。次に、1つの実案件に近いサンプルで、要件ID registry、受入trace、禁止trace、実装ログを揃えてエンドツーエンドで回す。最後に、Claude Code SkillとCI comment botを統合し、PR上で「失敗した要件ID」「最小反例」「推奨repair対象」「人間承認待ち」を返す。成功条件は、自然文再生成より修復率が明確に高いこと、同種ミス再発が減ること、反例がPRレビューで人間に理解されること。失敗条件は、FSL修正が広範囲化して局所化しない、要件IDやtrace作成の負担が大きすぎる、反例JSONが人間レビューに翻訳できないこと。投資前の反証プローブは、同じ自然文要件から自然文repair群とFSL反例repair群を並走させ、修復率・局所化・再発防止が改善しない場合は方向Aを主投資にしない。

方向Aの成果物は4つに絞る。第一にClaude Code Agent Skillで、`write_fsl -> fslc verify -> parse JSON -> repair -> rerun`を固定手順化する。第二に反例APIで、`requirement_id`, `trace_type`, `counterexample_steps`, `violated_guard`, `dead_reachable_path`, `suspected_missing_trace`, `repair_hint`, `guarantee_limit`を返す。第三にCI/PR差戻しで、ゲートというより「要件責任者の承認待ち証跡」を生成する。第四にベンチダッシュボードで、修復率、局所化、再発、要件ID欠落、forbidden欠落、reachable死経路を追う。ここで「CIが落ちたから直せ」だけにしない。overturn通り、CIは迂回されるため、PRコメントに要件責任者が承認・差戻しできる証跡を添える。

**方向B：意図ずれ監査台帳**

方向Bの狙いは、FSLを購買者向けに「正しさ保証」ではなく「意図ずれ監査台帳」として出すこと。対象はPM、開発統制、内部監査に限定する。支払主体は開発統制予算、内部監査/ITガバナンス予算、重要案件のPoC予算。責任部署はPMO、品質保証、開発統制、内部監査であり、開発チームは証跡生成の協力者に置く。非技術者向け証跡は、要件IDごとに「業務リスク」「受入条件」「禁止すべき経路」「死んでいる経路」「次アクション」「責任者」「承認状態」を示す台帳で、生JSONは添付資料に下げる。

最初の施策は、監査台帳レポートの型を作ること。1ページ目は案件単位のリスク一覧にする。項目は、要件ID、業務目的、検出種別、反例要約、想定損失、受入条件との関係、forbidden traceとの関係、reachable結果、実装ログ適合、判断者、次アクション、期限、保証限界。2ページ目以降は要件ID別の詳細で、`fslc` JSONの反例を自然言語の業務シナリオに翻訳する。たとえば「返金は安全性制約を破らないが、reachableで返金完了状態に到達不能」と出た場合、「返金業務が仕様上死んでいる。受入条件に返金完了traceを追加し、責任PMが期待経路を承認する」まで落とす。

この施策が効く因果は、購買者はLark+Z3、BMC+k帰納法、JSON反例そのものには支払わないが、監査で「どの業務リスクが未確認か」「誰が受け入れるか」「次に何を直すか」には支払うからである。DOGFOOD-10の境界欠落は、要件ID registryがないと欠落自体が見えないため、台帳の中心を要件IDにする。DOGFOOD-9/10/11の意図ずれは、内部整合では閉じないため、台帳では「保証済み」ではなく「人間確認が必要な欠落」として扱う。保証限界は正直に書くが、それを売り文句にはせず、各要件IDの判断欄に「FSLで検出できる範囲」と「業務妥当性の人間確認」を分けて置く。

順序は、まず既存DOGFOOD所見を3件の監査台帳サンプルに変換する。次に、PM/開発統制/内部監査に15分で読ませ、差戻し判断ができるかを検証する。次に、`fslc scenarios`と実装ログ適合チェックの結果を台帳へ自動投入する。最後に、PoCパッケージとして「2週間で重要要件20件を台帳化し、未確認リスクと差戻し項目を出す」形にする。成功条件は、非技術者が要件ID単位で承認・差戻し・リスク受容を判断できること、開発側が反例から修正へ戻れること、監査側が証跡として保存できること。失敗条件は、レポートが抽象的で次アクションに落ちない、生JSONの説明に終始する、保証限界が「結局何も保証しない」に見えること。投資前の反証プローブは、PM/監査担当に台帳だけを渡し、3つの判断、つまり承認、差戻し、リスク受容を要件ID付きで実行できるかを見る。できなければ方向Bの成果物設計を作り直す。

方向Bの成果物は、監査台帳レポート、要件ID registry、trace coverage表、risk-to-action summary、承認ワークフローである。要件ID別の見せ方は固定する。「REQ-17 返金完了」なら、業務意図、受入trace、forbidden trace、reachable結果、実装ログ結果、検出された欠落、反例の短い業務シナリオ、推奨アクション、判断者、期限、保証限界を1単位にする。`fslc` JSONは監査証跡として保存するが、購買者向け本文では「返金完了経路が仕様上到達不能」「境界値Xで禁止経路に入る」「REQ-12に対応するforbidden traceが未登録」のように翻訳する。

**A→Bの投資振り分け**

方向Aに強く寄せる条件は、実案件相当ベンチで自然文repairより修復率、局所化、再発防止が改善し、Claude Code Skill上で開発者が追加説明なしにrepairを回せること。特に、同一要件IDの再発が減り、反例JSONから修正箇所が狭まるなら、Aは開発基盤として成立する。

方向Bに寄せる条件は、Aの自動修復率が弱い、FSL修正が局所化しない、開発者がCIゲートを迂回する、要件ID/forbidden traceの妥当性確認が結局PMや監査に残る場合である。この場合、FSLの価値は自動修復ではなく、欠落・死経路・禁止経路を要件責任者に露出する監査台帳にあるため、Aは台帳生成エンジンに縮小し、Bを主商品にする。

初期投資配分はA:40、B:60にする。理由は、overturnによりCI/型システム単独路線は弱く、支払主体と責任部署が明確なのはPM/開発統制/内部監査側だからである。ただし、Aベンチで修復率と局所化が明確に勝てばA:60、B:40へ寄せる。逆に、repair改善が出ないが台帳で承認・差戻し判断が進むならA:25、B:75へ寄せる。

**捨てるもの**

構文教育は捨てる。FSLは人間向けDSL教材としてではなく、LLMと`fslc`のwrite→verify→repairを回す中間形式として扱う。docsの非専門家向けINTROは入口には使うが、収益の中心にしない。

汎用DSLも捨てる。対象をPM/開発統制/内部監査とAI開発基盤に限定し、acceptance/forbidden trace、reachable、要件ID透過、SLA、refinement、実装ログ適合の使い道に閉じる。

証明性能売りも捨てる。Lark+Z3、BMC+k帰納法は実装事実として必要だが、購買理由は「証明できる」ではなく、「仕様に書いた範囲の矛盾・反例・死経路・禁止経路・境界欠落を要件ID付きで露出し、人間が判断できる」ことである。

エージェント間仕様交渉プロトコルは本命にしない。需要が未証明であり、支払主体・責任部署・非技術者向け証跡が曖昧だからである。将来仮説としては残すが、Aのrepair基盤またはBの監査台帳で実需要が確認された後の派生に留める。

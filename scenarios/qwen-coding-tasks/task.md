# 課題: ローカル中位モデル qwen を「プログラミング」で活かせるタスクを発想する

tracefield はすでに qwen（ローカルの中位モデル, qwen3.6:27b）を *推論レンズ・反復の実行器官* として使えている。同じように、**ソフトウェア開発（プログラミング）の現場で qwen を有効に使えるタスク**を、できるだけ多く・具体的に発想したい。

## qwen（ローカル中位モデル）の輪郭（再説明は不要）

- **持ち出さない**: 完全オフライン／機密コード・データを外部 API に出さずに回せる。
- **安い・大量**: 単価が極小で rate limit がない。何千回も回すバッチ・常時稼働に向く。
- **低遅延・ローカルループ**: エディタ／CI／pre-commit に組み込んだ短い往復・即時フィードバックが効く。
- **境界が明確な変換が得意**: スキーマ・文法・定型に拘束され、入出力が閉じた機械的なタスク。
- **足場の中の実行器官**: 単発の賢さでなく、構造（隔離・反復・検証ループ・テスト/型のオラクル）が知能を担い、qwen はその小さく忠実な文脈を回す（tracefield 自身がこの一例）。
- **不得意（frontier に任せるべき）**: 新規アーキテクチャ設計・微妙な正しさ・広い文脈横断の深い推論。これらは候補にしない。

## 出力

各自、上の輪郭のうち **自分の担当する角度** から、qwen が *得意とする具体的なプログラミングのタスク* を多数発想せよ。

- 1 アイデア＝1 エントリ（type=observation）を **6〜10 個**。結論・優先順位・打ち手の比較はしない（最後に統合役が行う）。
- 各エントリは具体的に: **タスク名＋入出力＋なぜ qwen で足りるか（なぜ frontier が過剰か）** を 1〜2 文。
- 各エントリに **meta.kind** ＝そのタスクの種別を 2 語以内の名詞句で自分で命名（例: "コード変換", "テスト生成", "ドキュメント", "分類抽出", "レビュー補助", "定型生成"）。同種は同じ文字列を使う。
- qwen のスペックの再説明・抽象論はしない。**現場で動かせる具体タスク**を出す。

その後、全断片は半自動的にタスク種別テーマへ束ねられ、最終統合で「qwen を活かす代表タスク（上位）」が根拠とともに決まる。

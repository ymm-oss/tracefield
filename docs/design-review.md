# 設計レビュー — 実験計画書A の方法論的・運用的な穴

> **目的**: [`experiment-plan.md`](./experiment-plan.md) を実行・凍結する前に、結果を無効化しうる、
> または比較を歪めうる設計上の穴を洗い出す。各項目には **問題 / なぜ重要 / 提案 / 反映先** を付す。
> 反映先が `pre-registration.md` のものは、データ収集前に確定すること。
>
> **読み方**: 重大度は **Blocker（実行を妨げる/結果を無効化しうる）> Major（比較を歪める/解釈を曖昧にする）> Minor（運用上の注意）**。
> 計画書がすでに自覚している限界（§3.3 相互作用影響、§13 provenance 真正性）は、**自覚はあるが未解決**として扱う。

---

## Blocker

### DR-1. 反実仮想差分が「因果影響」と「実行ごとの分散」を区別できない
- **問題**: 接地真実（§3.1）は Condition A（汚染あり）と B（汚染なし）の出力差分。だが LLM マルチエージェント探索は seed・温度・入力順序を固定しても**完全には決定的にならない**（ツール呼び出し順、非同期、文脈依存の分岐）。A−B 差分は「汚染の因果影響」と「ベースラインの実行ごと分散」を**混同する**。
- **なぜ重要**: 主アウトカム（Impact Recall/Precision）の**分母そのものが汚染**される。差分の大半がノイズなら、Recall/Precision は測定対象を測っていない。
- **提案**: 汚染状態を固定したまま seed だけ変える反復（A vs A′、B vs B′）で**ヌル差分の分布**を推定し、「影響あり」を *ベースライン分散を超える変化* と定義する。代替として、固定分岐点からの介入（branch-point intervention）で経路を揃える。
- **反映先**: `experiment-plan.md` §3.1 / `pre-registration.md` §5（ヌル差分推定の反復回数・閾値）。

### DR-2. 接地真実の構成法が条件間で非対称（測定法バイアス）
- **問題**: ①「Aを除くと探索経路全体が変わる」（§3.3 が自覚）の影響は free-form 条件（C3/C4）で特に大きく、差分が *ほぼ全体* になり Precision の分母が崩壊する。②差分の項目型（candidate delta / frame revision proposal）は**半溶解性条件にしか存在しない**。つまり接地真実の「項目空間」が条件ごとに違う。
- **なぜ重要**: 反実仮想法が半溶解性（構造的で安定）には適用しやすく、free-form には適用しにくいなら、**測定法が比較対象に有利/不利**。これは「測りたいもの」と「測り方」の交絡で、cross-condition の Recall/Precision 比較が apples-to-oranges になる。
- **提案**: 接地真実を**条件中立な項目空間**（例: 正規化した claim / final recommendation / span）に射影してから採点する。candidate delta 等の条件固有型は内部診断に回す。接地真実の構成妥当性を条件ごとに検証し、崩壊する条件は主比較から除外するか別指標で扱う。
- **反映先**: `experiment-plan.md` §3 / §7 / `pre-registration.md` §2（item 正規化）, §5（条件間妥当性）。

### DR-3. 専門家ブラインドが出力フォーマットで破れる
- **問題**: §3.2 / §10.1 は条件名を伏せる。しかし出力**形式そのもの**が条件を露呈する（半溶解性は provenance ログ・candidate delta・packaging を持ち、free-form は生 transcript、固定Role は role 別構造）。専門家は形式から条件を即座に推測できる。
- **なぜ重要**: ブラインドが実質的に無効化され、確証バイアス対策（§10）が機能しない。
- **提案**: 採点前に全条件の出力を**共通フォーマットへ正規化**（懸念リスト + 推奨 + 影響主張、ログ類は剥がす）。正規化が不可能な指標は「ブラインド不能」と明示し、その限界を記録。
- **反映先**: `pre-registration.md` §8（正規化手順）。

### DR-4. 偽陽性対照（デコイ）が無く、Impact Precision を正直に測れない
- **問題**: §6 で注入するのは**すべて実際に汚染**された入力。良性だが疑わしく見える入力（benign decoy）が無い。
- **なぜ重要**: 「全部を影響ありと flag する」システムと「識別できる」システムを Precision 側で区別できない。clean 入力に対する誤警報率（false-alarm）が一切テストされない。
- **提案**: 撤回されない・有効なままの**デコイ入力**を同数程度混ぜ、これらを「影響あり」と誤判定する率を Precision 評価に組み込む。
- **反映先**: `experiment-plan.md` §6 / `pre-registration.md` §9（デコイ数）。

### DR-5. 検出力分析（power analysis）が無い
- **問題**: §14 は全パラメータ未定。だが反復回数の根拠となる**検出力分析が存在しない**。専門家採点は分散が大きく、8条件×複数指標で必要 n は大きくなりうる。
- **なぜ重要**: 検出力不足だと**ヌル結果が解釈不能**（真のヌルか、単に検出できなかったか判別できない）。失敗基準（§16）の判定が成立しない。
- **提案**: 主比較（DR-15 参照: C5 vs C4 を主とする）について想定効果量での検出力計算を行い、反復回数を逆算。パイロットで分散を実測してから本実験の n を決める。
- **反映先**: `pre-registration.md` §9 / §10。

## Major

### DR-6. 計算資源が条件間で非統制（C2 帰無仮説が潰せない）
- **問題**: C2（大規模固定Roleパネル）と C5（半溶解性）で、エージェント数・総トークン・LLM呼び出し回数・wall-clock が揃っているか未規定。
- **なぜ重要**: 半溶解性がより多くの agent-turn を使うなら、優位は「構造の良さ」ではなく「計算量の多さ」かもしれない。C2 の帰無仮説（「Roleを増やせば同等」）は**計算量を揃えて初めて潰せる**。
- **提案**: 計算予算（呼び出し回数 or トークン）を条件間で固定するか、コスト正規化した指標も併記。
- **反映先**: `pre-registration.md` §9。

### DR-7. アブレーション（C6/C7/C8）が「測定除去」か「挙動変化」か曖昧
- **問題**: 「半溶解性 − provenance」等は、監査層だけ外したのか、エージェントの**振る舞いまで変わる**のか不明。provenance を見ながら動くエージェントは、無いと探索自体が変わりうる。
- **なぜ重要**: 解釈が真逆になる。測定除去なら「provenance の追跡寄与」を、挙動変化なら「provenance がある前提の探索の崩れ」を測ることになる。
- **提案**: 各アブレーションで「探索過程は同一・監査層のみ差」か「能力ごと除去」かを明示し、どちらを意図するか確定。
- **反映先**: `experiment-plan.md` §4（C6–C8）/ `pre-registration.md` §9。

### DR-8. Containment / Reversibility がベースラインに存在しない機構で、cross-condition 比較が成立しない
- **問題**: §7.3 Containment と §7.5 Reversibility 分類は、C1/C3/C4 が**そもそも持たない機構**（durable state / quarantine / 分類器）に依存。にもかかわらず §7.3 は「差を見る対象は provenance/EGI の有無」と cross-condition を示唆。
- **なぜ重要**: 封じ込めを主張しないシステムに「封じ込め成功」を採点するのは不整合。これらは事実上**半溶解性内部の診断**に近い。
- **提案**: ベースラインに対する操作的定義（例: 「最終推奨に汚染主張が残存したか」なら全条件で測れる）を与えるか、内部診断として明示し主比較から外す。
- **反映先**: `experiment-plan.md` §7.3 / §7.5。

### DR-9. 接地真実2系統（反実仮想 vs 専門家裁定）の優先規則が未定
- **問題**: 反実仮想が「変化あり」、専門家が「依存していない（ノイズ由来）」と食い違う場合の確定ルールが無い（pre-reg §5 でも TBD）。
- **なぜ重要**: 2つの ground truth が並立したままでは Recall/Precision の分母が一意に定まらない。
- **提案**: 優先規則（例: 反実仮想を一次・専門家を上書き拒否権、または両者一致のみを「確実影響」とし片方のみは「疑い」に層別）をデータ前に固定。
- **反映先**: `pre-registration.md` §5。

### DR-10. item の定義と重複カウント
- **問題**: 単一の汚染影響が「変化した claim」「変化した trace」「変化した recommendation」として**多重に**現れうる（§3.1 の5型）。各を別 item と数えると分母が膨らむ。
- **なぜ重要**: 同じ影響を多階層で出すシステムを不当に有利/不利にする。Recall/Precision が比較不能になる。
- **提案**: 因果グルーピング/重複排除の規則を定め、「1つの影響 = 1 item」へ正規化。
- **反映先**: `pre-registration.md` §2。

### DR-11. 「影響追跡」と「撤回後の信念改訂」が混在している
- **問題**: 汚染 B は*撤回された証言*、C は*ポリシーで無効化*。これらの本質は「Aが触れた全てを見つける（影響追跡）」ではなく「Aが偽と判明した後に正しく改訂する（belief revision）」。主指標 Impact Recall は前者を測るが、Containment/Repair は後者寄り。
- **なぜ重要**: 何を成功とするかが汚染種別で食い違い、指標と現象が対応しない恐れ。
- **提案**: 汚染種別ごとに「影響追跡課題」か「改訂課題」かを分類し、対応指標を明示。
- **反映先**: `experiment-plan.md` §6 / §7。

### DR-12. 汚染入力の注入タイミングが非統制因子
- **問題**: A/B/C を探索の**いつ**注入するか未規定。早期注入ほど波及が大きい。
- **なぜ重要**: タイミングが交絡変数になり、条件間/反復間で揃わないと差が注入位置由来になりうる。
- **提案**: 注入点を固定要因として設計（固定 or 体系的に変える要因として明示）。
- **反映先**: `experiment-plan.md` §6 / `pre-registration.md` §9。

### DR-13. 連言的成功基準（§15 の6条件同時）の検出力コストが未分析
- **問題**: §15 は6条件を**同時に**満たすことを成功とする連言基準。
- **なぜ重要**: 連言は偽陽性に強い一方、各条件に検出力ゆらぎがあると**偶然に全成立する確率が極端に低く**、真に有望でも「失敗」と出やすい。joint なエラー率/検出力が未分析。
- **提案**: 主要成功条件（必須）と補助条件（望ましい）を層別し、主比較を1つに絞る（DR-15）。連言の joint power を見積もる。
- **反映先**: `pre-registration.md` §10 / §11。

### DR-14. 単一シナリオでは一般化できず、汚染も流用できない
- **問題**: 具体例は「企業向けAIアシスタント」1系統のみ。§14 シナリオ数 TBD。汚染 A/B/C はこのシナリオに密結合。
- **なぜ重要**: 1シナリオの結果はシナリオ特有の癖に支配され、枠組みの一般性を示せない。
- **提案**: 最低でも領域の異なる複数シナリオを用意。シナリオごとに介在的死角と汚染を設計（設計コスト大として明記）。
- **反映先**: `experiment-plan.md` §5 / §6 / `pre-registration.md` §9。

### DR-15. 主比較・主アウトカムが1つに絞られていない（多重性）
- **問題**: 8条件 × 主5+副5 指標で多数の検定。§2 の主仮説は C5>C4 を示唆するが、計画は単一の主比較×主指標を明示指定していない。
- **なぜ重要**: garden of forking paths。事後に都合の良い比較を「主」と呼べてしまう。
- **提案**: **主比較=C5 vs C4、主アウトカム=Impact Recall**（例）を1つ確定し、他は副次/探索的と宣言。多重比較補正を規定。
- **反映先**: `pre-registration.md` §1 / §2 / §10。

## Minor / 運用

### DR-16. Novel/Interstitial Concern Count が正規化されておらず gameable
- **問題**: 「baseline が出さなかった懸念数」は冗長なシステムほど生数が増える。「妥当」「介在的」の判定手順も未操作化。
- **提案**: 重複排除した *ユニークかつ妥当* な懸念で数える/出力量で正規化。Interstitial は「パネルが各懸念に必要 role を割当て、単一 role で所有不能なもの」と手続き定義。
- **反映先**: `experiment-plan.md` §8.1 / §8.2 / `pre-registration.md` §6。

### DR-17. Packaging Loss / Survival of Insight の参照基準と回帰問題
- **問題**: 「生 trace の重要な意味」を誰が定義するか不明（無限後退）。両指標は C5/C6/C7 のみで成立（C8 は評価を外すだけ）。
- **提案**: 参照を「事前に専門家が raw trace から抽出した *重要点リスト*」に固定し、それの残存率で測る。両指標は内部診断であり対戦成績でないことを明示（Packaging Loss は明記済、Survival of Insight も同様に）。
- **反映先**: `experiment-plan.md` §8.4 / §8.5。

### DR-18. 曖昧構成概念の評価者間信頼性（IRR）が低くなりやすい
- **問題**: 「妥当な懸念」「介在的」「重要な意味の喪失」は最も IRR が低くなる種類。§10.3 に閾値が無い。
- **提案**: 本実験前に**パイロットで IRR を実測**しルーブリックを改訂。閾値未満時の手続き（再訓練/第三者裁定/項目除外）を pre-reg §8 で確定（既にフィールドあり）。
- **反映先**: `pre-registration.md` §8（閾値・パイロット）。

### DR-19. 「最小実験」のコストが過小評価されている
- **問題**: §17 は「最小実験」と称すが、実体は **4種のオーケストレーション系 + 反実仮想インフラ + 採点パイプライン + 複数領域の専門家パネル + ホールドアウト専門家**。最小ではない。
- **提案**: 段階化。**MVP（1シナリオ・1汚染・C4 vs C5・自動プロキシ指標）→ パイロット（IRR/分散実測）→ 本実験**。各段で中止判断点を置く。
- **反映先**: `experiment-plan.md` §17（実施段階） / `pre-registration.md`（パイロット計画）。

### DR-20. 専門家の確保と領域適合
- **問題**: タスクは法務/セキュリティ/UX/ガバナンスにまたがる。妥当性判定には本物の領域専門家が要り、ホールドアウト（§10.4）でさらに倍。
- **提案**: 必要専門領域と人数を pre-reg §8/§9 で確定し、確保可能性を先に検証。代替として領域別に分業採点。
- **反映先**: `pre-registration.md` §8 / §9。

## サマリ — 優先対応順

1. **接地真実の妥当性**（DR-1, DR-2, DR-9, DR-10）— 主アウトカムの分母が成立するかの生命線。これが崩れると他は無意味。
2. **比較の公平性**（DR-3 ブラインド, DR-6 計算量, DR-7 アブレーション, DR-8 ベースライン不在機構）。
3. **測れる設計**（DR-4 デコイ, DR-5 検出力, DR-15 主比較, DR-13 連言基準）。
4. **段階化と現実性**（DR-19 MVP化, DR-18 IRRパイロット, DR-14 複数シナリオ, DR-20 専門家確保）。
5. **指標の頑健化**（DR-11, DR-12, DR-16, DR-17）。

> 推奨: まず DR-19 の **MVP（C4 vs C5・1汚染・自動プロキシ）** を組み、その上で DR-1/DR-2 が現実にどれだけ深刻かを実測してから、本実験のパラメータ（pre-registration）を凍結する。

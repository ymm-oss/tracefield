# 設計 — クラスタ（エージェント集合の単位）とクラスタ間接続

> ユーザー定義（2026-06-10）: **「tracefield ではエージェントの集合が複数現れて、それをクラスタと呼び、
> それらが繋がることが行われる」**。本書はこの概念を固定し、既存の実証知見から接続の設計原則を導く。
> 関連: [`design-agent.md`](./design-agent.md)（主体）、[`design-reference.md`](./design-reference.md)（場）、[`overview.md`](./overview.md) §0。

## 0. 用語の区別

- **クラスタ（本書）** = エージェントの集合＋共有場（Reference store＋共有手続き＋蓄積来歴）。半溶解チームの単位。
- **クラスタリング（測定）** = 主張の同一性を定める計器の操作（glossary C節）。別概念。

## 1. 定義

```
クラスタ = { エージェント集合（各自: 私的文書・私的メモリ・偏り）,
             共有場（Reference store: chunks・entries・手続き・撤回状態）,
             クラスタ自身の同一性（名前・関心・蓄積された判断の癖） }
```

現行実装は**単一クラスタ**（1 シナリオ = 1 チーム = 1 store.jsonl）。本設計はその複数化と接続を扱う。

## 2. 核心 — 再帰性（クラスタは上位レベルのエージェント）

[`design-agent.md`](./design-agent.md) の定義「**エージェント＝状態と手続きを行える主体**」は、クラスタにそのまま成立する:

| Agent の構成要素 | クラスタでの対応物 |
| --- | --- |
| 状態（私的） | クラスタの store（外に出していない entries・来歴） |
| 手続き | クラスタの共有 procedure 群（方法論・規約） |
| 器官（LLM） | **チーム全体**（クラスタの「推論」は内部の半溶解協働で実行される） |
| 偏り | クラスタの専門性・文化（蓄積判断の癖 ＝ store が育てる来歴の偏り） |
| 外部化（absorb） | クラスタが上位の場へ要約・決定を projection する |

→ **半溶解性はスケールフリー**: agent ⊂ cluster ⊂ cluster-of-clusters。
溶解ダイヤル（k_s / k_p）と「半」の動作点が**各レベルに**存在する。

## 3. 接続の機構（Projection / Absorption の本来の意味）

クラスタ間接続 = **場から場への射影と吸収**（実験計画書の原語彙がここで全義を得る）:

1. **entry の輸出入（基本）**: クラスタA の entry をクラスタB の store に absorb。
   `meta: %{source_cluster: "A", source_id: "e13"}` を必ず付け、**クロスクラスタ provenance** を保つ。
   引用は「B 内の写し entry の id」を経由（写しが原本を指す）。
2. **境界エージェント**: 2つの store を perceive する ambassador（人間組織の兼務に相当）。
3. **メタ場（field of fields）**: 各クラスタが要約・決定を外部化する上位 Reference。
   クラスタが**エージェントとして**上位の半溶解に参加する（§2 の再帰の実装形）。

## 4. 実証済み知見からの設計原則（予測）

単一クラスタで実証した法則は、クラスタ間でも成立すると予測する（同型性）:

- **§11（用量反応）→ 接続も「半」であれ**: クラスタ間を深く繋ぐ（store 全共有）と**クラスタの偏り＝専門性・文化が均質化**する。
  接続は選択的 pull・引用付き・撤回可能に限る。全社で1つの巨大 store は完全溶解＝統治不能かつ没個性。
- **§13（ファネル）→ クラスタ間 serve が律速**: どの entry を相手クラスタに見せるかの方策（輸出の選択）が便益を決める。
- **§14（契約の自覚）→ クラスタも構造を知る必要**: 「この輸入 entry は他クラスタの私的知識への唯一の窓」という
  自覚をプロンプト/手続きで与えなければ、輸入物はただの文脈としてエコーされる。
- **撤回の越境（統治の拡張）**: クラスタA で撤回された entry の写しを持つクラスタB では、
  写し経由の閉包が隔離対象になる ── **案件横断・チーム横断の「要件変更/知見訂正の波及管理」**。
  これは行き先①②両方のキラー機能（例: 案件A で否定された前提に依拠した案件B の提案を即特定）。

## 5. 行き先へのマッピング

- **行き先①（AI駆動開発）**: リポジトリ/領域ごとにクラスタ（frontend / backend / security / QA…）。
  共有 ADR・横断要件はメタ場に。要件変更はクラスタ境界を越えて閉包隔離。
- **行き先②（コンサル）**: 案件ごとにクラスタ＋全社ナレッジのメタ場。
  案件の学び（匿名化済み）をメタ場へ projection → 他案件が吸収（出所つき）。クライアント情報の訂正は写し先まで波及。

## 6. 実装への距離 → **最小実装済み（brief-18）**

store は既に per-scenario のファイル（`store.jsonl`）なので、最小実装は近い:
- ✅ `Reference.export/import`（author="src/orig" 合成・source_cluster/source_id meta・バッチ内引用再マップ・バッチ外は meta.unresolved_citations・冪等・embedding 再利用・persist 対応）
- ✅ `Reference.propagate_retractions`（source の retracted/superseded を写しへ反映 → 閉包隔離。persist で再起動後も保持）＋ `mix tracefield.bridge`（--export/--sync/--demo）。
- ✅ メタ場 `Tracefield.Meta`（publish=被引用数順の代表選択/ids 明示、discover=embedding 類似＋exclude_cluster、pull）。
- ✅ **有機的接続の最小形（brief-19）**: `Reference.subscribe`（status イベント配信）＋ `Bridge.Link`（購読→自動 propagate）＋
  `Tracefield.Field`（supervision tree で複数クラスタ＋META＋links:auto が**同時に生きる**）。
  デモ: B が embedding 発見で A の知見を pull → 判断 → A の撤回が **META 経由で B まで無人伝播** → 依存判断隔離。
  多段 provenance は `meta.source_chain`（A/e1 → META/e1）で保持。
- 残: クラスタの自動生成（自己組織化）・ノード越え分散・Jido AgentServer 全面採用。

## 7. 未解決の設計問題（正直に）

- **輸出の選択方策**（何をクラスタ外に出すか）: 機微情報の境界（SEC 判断の越境版）と便益のトレードオフ。
- **id 空間**: 写し方式は単純だが、同一 entry の写しが多段に増殖する（provenance は保たれるが重複管理が要る）。
- **クラスタの偏り測定**: agent 用の diversity/collapse 計器をクラスタ間に持ち上げる（store 間の埋め込み距離）。
- 同期 vs 非同期接続（push 通知か、次回起動時 pull か）。

## 8. 垂直伝達 — クラスタの偏りはメンバーに伝わるか（2026-06-10 追加）

**現状の正直な答え**: 部分的に3経路で伝わる ── 共有手続き（強・毎ターン注入）、charter=task.md（強・anchor）、
store 検索（**弱・k_s 件のサンプル窓のみ**）。専用の「文化」チャネルは無く、エージェント私的メモリは
**自分の判断のみ**蓄積するため、クラスタが育てた判断文化は新参メンバーへほぼ継承されない。

**本質的な緊張（第3の垂直ダイヤル）**: 伝達を強めるほどメンバーが均質化する（§11 の同型）。
- 伝わらなすぎ → 文化なきチーム（ただの寄せ集め・クラスタの偏りが実在しない）
- 伝わりすぎ → メンバー均質化（個の偏り消失 ── クラスタ内の多様性が死ぬ）
- **健全 = charter 整合（高）× メンバー間多様性（高）の同時成立**

**設計（v1 → 将来）**:
1. **計器が先**（v1）: `culture transmission` 計測 ── 整合 alignment = メンバー公開 entries と charter の
   embedding 類似の平均（著者別も）、温存 = 既存のメンバー間 diversity。両方を常時表示し動作点を可視化。
2. ✅ **蒸留（distillation）**（brief-21 実装済）: `Culture.distill`（被引用順の抽出蒸留 or LLM 蒸留、版管理・先代 supersede）→
   `:house_view` entry（**citations=蒸留元＝文化の provenance**）→ ideate `--distill` で次 run の全メンバーへ**有界注入**
   （「踏まえつつ、自分の偏りからの異見も歓迎」の文言で第3ダイヤルを保護）。
   **統治がテスト済**: 元判断の撤回 → house view が閉包に入り失効 → 再蒸留は撤回分を含まない（文化の自動失効・更新）。
   新メンバーの受け入れ＝最新の蒸留を渡すこと。人間組織が文化を「歴史の再生でなく規範の要約」で継承するのと同型。
3. 析出した新生クラスタは charter＋手続き＋種 store で**初期文化**を持ち、メンバーの私的メモリは空
   ── 文化はメンバーが作り、文化がメンバーを形成する**双方向**が出生時から始まる。

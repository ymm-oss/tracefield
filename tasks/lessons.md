# Lessons

## 説明は analogy 先行にする（コード行・英語用語で固めない）
- 2026-06-21: tracefield の撤回 / aggregate を「わかりやすく教えて」と頼まれたのに、最初の2回をコード行参照＋英語ジャーゴン（defeater / tally / invariant）＋ASCII図で密に説明し、「まったくわからない」と言われた。3回目に裁判のたとえ（証人＝レンズ、つっこむ人＝verify、反論ごとに裁判官1人＝adjudication、書記＝aggregate）へ置き換えたら即「わかった」。
- 教訓: 「わかりやすく」と言われたら、まず現実の役割に写像した具体的なたとえ話で芯を伝える。コード行・英語用語・図はその後、求められたら。抽象度の高い機構ほど analogy-first。

## 実験のプロンプトに「答え」を漏らすな（2026-06-21・ユーザー指摘×2で確立）
- オーケストレーション vs 単一の比較で、**勝たせたい側のプロンプトに測定対象(正解)を埋め込む**過ちを2回繰り返した:
  1. **de-risk 足場**: uncued 衝突タスクで単一(Arm S)の足場に「不可逆な損害を分析せよ/de-risk を先頭に」=*発見すべき overturn そのもの*を渡し、偽の「edge なし」。
  2. **レンズに具体ケース**: テスト生成でレンズに「接する境界(end==start)・1だけ離れた最小ケース・順不同・ネスト」=*mutant が突く失敗様式そのもの*を列挙、単一には観点ゼロ → 偽の「発見 edge(5/5)」。
- 教訓: プロンプトは測定器の一部。一方に正解を書くと「能力差」でなく「私がどちらに答えを教えたか」を測る。**両腕に同じ情報を渡す(対称性)**。オーケストレーションのレンズ/足場に書いた観点は単一にも渡した変種(arm-W=観点を1文脈に与える)を必ず作る。レンズ/足場は**汎用(タスク非依存の角度・方法)**に保ち、タスク固有の正解ケースを列挙しない(具体はモデルが導く)。ベースラインは neutral。**測定対象(mutant/overturn/植え込み)を、どちらのプロンプトにも文字列として出さない(grep 自己点検)**。継続的にこの罠に落ちた=デフォルトで警戒。

## 同一モデルの裁量 verify/adjudication は「収束」を捏造する（2026-06-22・ユーザー指摘で発覚）
- codegen-econ-reread の統治フローが「帰責可能性=希少資源」に収束したと報告したが、ユーザーに「analysis 以外の codex は自分の志向する方向を選択的に評価してないか？」と問われ調べると: verify(FALSIFY/COUNTER, `fixed` count=2)も adjudication も同じ codex で、**反証6発が全て codex の attractor から外れたコスト枠4主張に集中、帰責枠7主張は一度も攻撃されず無傷生存**。「8生存」は反証を生き延びたのでなく*選ばれなかった*だけ。私の「生存者は帰責を指す」総合は選択バイアスの産物だった。
- 教訓: **同一モデルの*裁量*refuter(agent が攻撃対象を選ぶ)は、モデルの好む主張を見逃し、外れる主張だけ潰す**。生存集合を「検証された結論」と読むな。攻撃の**網羅と対称性**を必ず確認する(主張ごとの被攻撃回数を数える=計器)。是正は (1)`per_input` で全主張に1反証強制(裁量を消す) (2)対立2極の**ディベート**で motivated・対称な攻撃にする＋反 halo 審判 (3)真の対照は stage 別 `organ` の異種モデル。同一モデルである限り深層の共有 prior は残る=「構造の発見」と「prior の復唱」は同一モデルでは完全には切り分かない。skill 反映済(SKILL「同一モデルの…選択バイアス」/patterns「7. 2極ディベート」)。

## 単一 LLM の「組織化」段(圧縮器/統合器)も prior=定番へ再収束する（2026-06-23・拡散アイデア出し qwen-coding-tasks）
- 反復ループ(scatter⇄deepen)で*生成*は定番(コード片の構文変換)を破り新領域(運用/機密/配布/protobuf/codemod)を出せたのに、最終段の単一 LLM **NORM が 48→3 に圧縮し新領域 30+ を黙って落とし**、SYNTH 上位は定番へ逆戻り。L3(裁量 verify の収束捏造)と同型が*組織化段*に再発。NORM を撤去し**決定論の無落としクラスタ**(`max_clusters` を distinct kind 数以上にして `other_sources` に隠さない)へ替えると 3→31 在庫が可視化され上位も広がった。だが**唯一残る単一統合器 SYNTH は完全在庫(31)を前にしても冒頭8クラスタへ自発アンカー**(出力≈400tok≪max_tokens=6000＝トークン制約でない)。
- 教訓: **生成を束ねる*いかなる*単一 LLM 段(NORM 圧縮器でも SYNTH 統合器でも)も prior=定番を再賦課し稀信号を落とす**。北極星「中央 LLM 合成を置くな/機械集約せよ/黙って落とすな」は verify/adjudication だけでなく拡散の cluster/synthesis にも効く([[thought-chain-vs-task-chain]])。fix 梯子=(1)落とし得る単一 LLM 圧縮器を使うな→決定論*無落とし*クラスタ(落としを*隠す*NORM より、落としを*露出*させ「31在庫の8しか使っていない」と証明可能にする決定論クラスタが上)(2)統合器に発散マップ(deepen)＋被覆契約=緩和のみ(3)原理解=**領域別に隔離した統合器→機械連結**。ただし(3)は現エンジン未対応＝隔離モード(`per_input`/`per_cluster`)が全入力をシャードし「1セレクタでスケール＋他を共有文脈として渡す」ができない(`scale_by` 本来の役割=未実装の設計ギャップ)。根拠 `docs/findings-bet2-overturn.md`「アイデア出し(拡散)」。
- 追記(組織化器3種を同一upstreamで実測): **debate >> 単一SYNTH > MAGI型3体多数決**。MAGI(同一モデル3評議体→多数決)は9票全て定番に着地・新領域に投票ゼロ・少数を small_sources へ黙殺＝**多数決は同一priorを「合意」に偽装し稀信号を落とす**(Condorcet独立性違反)。debate だけが prompted opposition で新領域を考慮に入れ、おまけに「定番は既存ツールが無料で解決済=低価値」を暴いた。決め手は数でなく**集約原理**: 多数決(合意)< 可謬性(1反証がN合意を破る=`aggregate_verdicts` native)。MAGI/多数決が正当なのは異種substrate×独立誤り×故障耐性(TMR)の別レジームのみ＝discovery には使うな。

## 根本原因は「reader を grep し実データの型を見る」前に理論化するな（2026-06-23・自己検証で発覚）
- debate の meta.refutes 伝播ギャップを2回誤診断した: ①「プロンプトで ADJ に meta.refutes を出させれば直る」(再走で失敗) ②「per_input が対象 id を見れない＝`scale_by` と同族の隔離メタ伝播ギャップ」(要エンジン拡張と findings/skill に記録)。**実際は** reconcile は verdict の citations→rebut.meta.refutes で正しく対象解決済で、唯一の不具合は `reconcile_overturned` が `meta.refutes` を*配列でしか受けない*こと(agent はスカラ文字列でも出す＝実測 FRONTIER str/CONV list)＝単純な parse 不寛容。store.rs:199 の1行を grep し、データで refutes が str/list 混在と見れば即判明した(修正は両受け1箇所＝debate3 で UNACTIONED 2→0)。
- 教訓: 機構の不具合は**まず唯一の reader を grep・実データの値型を見てから**診断する。アーキテクチャ的大仮説(隔離/`scale_by`/「モデルが見れない」)へ飛ぶ前に最小の事実(誰が読むか・値の型は何か)を確認。誤診断のまま findings/skill に「要エンジン拡張」と書くと負債になる(今回 falsified を訂正で消す羽目に)。[[thought-chain-vs-task-chain]] 同様、主張は接地してから記録する。

## ユーザーの開口部のフレーム(提示された梃子)を疑わず大作で応答するな（2026-06-23・「問題は…じゃない」で発覚）
- 「毎週MTGを処理、過去処理を参照できれば情報が多層になり高精度化する→そのためメモリが要る。現Storeをそのまま使えばいい?」と問われ、(1)コード接地でStore再利用可(永続/resume/seed重複回避)を詳述 (2)実痛点のブレインダンプを受け「判断Storeが正準/スライドは投影/直交foresightレンズ群」の大きな再設計を提示。直後に「問題は継続的にメモリーとして利用できるかじゃない」と*開口部の仮説ごと*否定された。ユーザーは梃子(=メモリ/想起)自体が的外れと既に気づいており、真のボトルネックは曖昧多状態の解釈+先読みだった。
- 教訓: 「Y のために X(梃子)をすればいい?」型の問いは、X を精緻化する前に**Y が本当のボトルネックか／X が本当の梃子か**を短く突く。初期フレームは検証対象であって前提でない。特にユーザーがブレインダンプ中は巨大な構造化ソリューションが過剰射撃になりやすい——まず問題を1-2文で写し返して合意(reflect-and-confirm)、それから設計。来歴(memory)も初期フレームに引きずられるので、撤回が出たら明示訂正する。
